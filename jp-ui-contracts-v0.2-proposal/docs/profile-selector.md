# profile selector

Use this when you are not sure which profile should be your starting point.

---

## Quick questions

### Q1. Is long-form reading the primary activity?
- Yes → start with `media`
- No → go to Q2

### Q2. Are forms, labels, settings, and repeated operations the core of the screen?
- Yes → start with `saas`
- No → go to Q3

### Q3. Is the screen primarily for technical explanation, help, or documentation?
- Yes → start with `docs`
- No → go to Q4

### Q4. Is high-density scanning of metrics, cards, tables, and filters the main activity?
- Yes → start with `dashboard`
- No → start with `base`

---

## Notes

- `base` is not the final answer. It is the safest neutral starting point.
- If your UI mixes multiple modes, pick the primary activity first, then add overlays later.
- Do not create a new top-level profile too early. Try an overlay or validator rule first.

---

## Practical heuristic

- Reading comfort first → `media`
- Operational stability first → `saas`
- Prose + code coexistence first → `docs`
- Scanability under density first → `dashboard`
